# 常用网址页面设计

日期：2026-04-08

## 目标

在当前 Vue 工具站中新增一个“常用网址”页面，用于展示和管理从 `bookmarks_2026_4_8.html` 中导入的 `devTool` 书签。

范围限定如下：

- 只导入并管理 `devTool` 分组中的书签
- 页面不体现 `devTool` 分组概念
- 支持搜索、快速打开、新增、删除
- 支持把修改后的数据直接写回当前 GitHub 仓库默认分支
- GitHub token 由用户在页面中手动输入，并可保存在 `localStorage`

## 现状与约束

- 项目是 Vue 3 + Vue Router 的静态工具站，没有后端
- 站点通过 GitHub 自动构建部署，适合继续保持纯前端架构
- 浏览器端不能把 GitHub token 写死在代码中，只能由用户运行时提供
- 初始书签来源为本地导出的 Chrome 书签 HTML 文件，已确认 `devTool` 中共有 47 条链接

## 方案选择

### 方案 A：独立 JSON 数据文件 + 前端直连 GitHub Contents API

将书签数据独立保存为仓库内的 JSON 文件，页面运行时读取该文件作为初始数据，用户在页面中进行搜索、增删后，再通过 GitHub API 把新的 JSON 内容提交回默认分支。

优点：

- 数据与页面代码分离，仓库 diff 清晰
- 不需要新增后端或部署额外服务
- 保持当前静态站部署方式不变

缺点：

- 浏览器端需要直接请求 GitHub API
- GitHub token 需要用户自行输入并在本地保存

### 方案 B：直接改源码中的常量数据

把书签写进某个 `.js` 或 `.vue` 文件中，由页面直接读取和更新该源码文件。

优点：

- 实现路径看起来更短

缺点：

- 数据与代码混杂，不利于长期维护
- 仓库 diff 可读性差
- 提交冲突处理更差

### 方案 C：新增后端代理保存

由前端调用后端，由后端持有凭据并提交 GitHub。

优点：

- 安全性最好

缺点：

- 明显超出当前仓库边界
- 增加额外运维和部署复杂度

结论：采用方案 A。

## 页面设计

### 入口

- 首页新增一个工具卡片“常用网址”
- 路由新增独立页面，例如 `/bookmarks`

### 布局

页面采用双区工作台布局：

- 左侧操作区：返回首页、搜索说明、新增表单、GitHub 配置、保存状态
- 右侧结果区：网址卡片网格

选择这个布局的原因是：

- 47 条书签适合在右侧用卡片网格展示
- 搜索、新增、保存三个高频操作需要稳定位置
- 比顶部工具条布局更利于高频管理

### 网址卡片

每个卡片包含：

- 标题
- URL
- “打开”按钮，使用新标签页跳转
- “删除”按钮

可选增强：

- 展示域名摘要
- 超长 URL 截断显示但保留完整跳转

### 搜索

搜索框放在左侧操作区顶部附近，输入后即时过滤右侧结果。

匹配规则：

- 同时匹配标题和 URL
- 大小写不敏感
- 不区分是否完整匹配

### 新增

新增表单至少包含：

- 标题输入框
- URL 输入框
- “新增”按钮

校验规则：

- 标题必填
- URL 必填
- URL 仅接受 `http` 或 `https`

新增成功后：

- 新网址立即出现在当前页面列表
- 保持为“未保存到 GitHub”的脏状态

### 删除

- 每张卡片提供删除操作
- 删除后立即从当前页面列表移除
- 删除同样只修改当前页面状态，直到用户点击“保存到 GitHub”

## 数据设计

### 初始数据文件

新增独立数据文件：

- `src/data/bookmarks.json`

文件内容保存导入后的书签数组。每条记录建议包含：

- `id`：稳定标识，优先使用可重复生成的简单 ID
- `title`：书签标题
- `url`：书签 URL

不保留 HTML 中的原始 `ADD_DATE`、`ICON` 等字段，除非实现时发现确有价值。

### 本地状态

页面运行时维护：

- 当前书签列表
- 搜索关键字
- GitHub 配置
- 当前保存状态
- 错误信息

GitHub 配置保存在 `localStorage`，建议包含：

- token
- repository，例如 `owner/name`
- branch，默认使用仓库默认分支

如果实现时能可靠推断仓库名，可预填；否则允许用户手动输入。

## GitHub 写回流程

页面点击“保存到 GitHub”时执行以下流程：

1. 校验 GitHub token 和仓库信息是否完整
2. 请求 GitHub API 读取目标文件当前内容和 `sha`
3. 生成最新的 JSON 内容
4. 调用 GitHub Contents API 更新 `src/data/bookmarks.json`
5. 提交到默认分支，提交信息使用固定格式，例如 `feat: update bookmarks`
6. 成功后更新页面内的“最后保存成功”状态

## 安全与风险处理

### Token 处理

- token 由用户手动输入
- token 输入框默认掩码显示
- 提供“记住到本地”和“清除 token”能力
- 不把 token 发往除 GitHub API 之外的服务

### 错误处理

需要明确提示的失败场景：

- token 缺失
- token 无权限或仓库名错误
- 网络失败
- 远端文件 `sha` 发生变化，导致提交冲突
- JSON 内容编码或提交失败

冲突时提示用户刷新页面重新拉取远端数据后再保存，不在当前版本内实现自动合并。

## 组件与职责拆分

推荐拆分如下：

- 页面视图：负责整体布局、交互编排、保存状态展示
- 数据工具模块：负责解析初始数据、规范化书签、生成待提交 JSON
- GitHub API 工具模块：负责读取文件、更新文件、处理 API 错误

这样可以把页面交互和 GitHub 请求逻辑分开，降低后续维护成本。

## 测试与验证

当前项目没有现成测试框架，因此本次至少保证以下验证：

- 初始 `devTool` 数据正确导入为 JSON 文件
- 搜索可以同时命中标题和 URL
- 新增时标题和 URL 校验正确
- 删除会即时更新当前列表
- GitHub 保存请求在参数完整时可正确构造
- GitHub 保存失败时页面会给出明确提示
- 构建通过
- lint 通过

如果实现时发现引入完整单元测试成本过高，则优先保证模块化和可验证的手工测试路径，并保留后续补测试的空间。

## 非目标

本次不包含以下内容：

- 管理其他书签分组
- 页面内编辑已有书签
- 拖拽排序
- 批量导入新的 HTML 书签文件
- 多用户并发协作处理
- 自动合并远端冲突

## 实施结果预期

完成后，用户可以在当前工具站中直接：

- 搜索现有常用网址
- 快速打开目标网址
- 新增网址
- 删除网址
- 使用页面中输入的 GitHub token，把最新地址列表直接提交回仓库默认分支

该能力与当前静态站部署方式兼容，不引入额外后端。
